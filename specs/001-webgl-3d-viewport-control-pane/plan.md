# Implementation Plan: WebGL 3D Viewport + Control Pane Shell

**Branch**: `001-webgl-3d-viewport-control-pane` | **Date**: 2026-04-28 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/001-webgl-3d-viewport-control-pane/spec.md`

## Summary

Establish the foundational application shell for Frameer 3D: a GPU-accelerated Three.js
viewport (left 2/3) and a collapsible Tweakpane control pane (right 1/3). Layout state
is persisted in `localStorage`. A `SceneManager` aggregate provides the stable extension
API through which future workflow elements inject `THREE.Object3D` subclasses into the
scene. `ControlPane` provides the shared Tweakpane `Pane` + `@tweakpane/plugin-essentials`
that workflow elements use to register their parameter UIs. OpenCASCADE.js (WASM) is
loaded at startup but not used in this shell — it is the geometry kernel for all future
workflow elements (frame parts, rings, mesh-cell lattices, n-gons, keeper bars, tie
connectors, vent frames) that port the `blender-mcp-tooling` Python/Blender CAD toolkit.

## Technical Context

**Language/Version**: TypeScript 5.5 (strict mode — all flags in `tsconfig.app.json`)  
**Primary Dependencies**: `three` (Three.js, MIT), `opencascade.js` (OCCT WASM, LGPL), `tweakpane` (MIT), `@tweakpane/plugin-essentials` (MIT), Vite 6, Vitest 4 + vite-plugin-doctest, pnpm 10  
**Storage**: `localStorage` only — key `frameer3d.v1.layout` (paneWidth, paneCollapsed)  
**Testing**: Vitest + jsdom; `SceneRenderer` interface seam for Three.js GL boundary; Tweakpane folders testable in jsdom; 98%+ coverage required  
**Target Platform**: Browser (Chrome/Firefox/Safari latest); WebGL 2 required; COOP/COEP headers set for WASM SharedArrayBuffer  
**Project Type**: Single-page web application (local-first, no backend, no service workers)  
**Performance Goals**: 60 fps render loop (`THREE.WebGLRenderer`); ≤ 1 layout reflow per drag event; HiDPI via `renderer.setPixelRatio(devicePixelRatio)`  
**Constraints**: No network requests; no raw GL shader management (Three.js owns the context); OCCT WASM loaded async at startup  
**Long-term domain**: Port of `blender-mcp-tooling` (Python) — 10 parametric part families (frames, rings, mesh lattices, n-gons, clusters, repeat grids, Boolean cuts, keeper bars, tie connectors, vent frames)  
**Scale/Scope**: Single-user local app; initial scene = `THREE.AxesHelper`; `SceneManager` + `ControlPane` extension APIs for future workflow elements

**Structure Decision**: Single-project web app. Tests co-located with source files per
AGENTS.md convention. The `viewport/` domain is the only one that directly touches
`THREE.WebGLRenderer`; it is tested via the `SceneRenderer` interface seam. The `occt/`
domain isolates all WASM concerns behind `GeometryKernel`. The `controls/` domain
isolates Tweakpane instantiation — workflow elements only receive `ControlPaneApi`.
